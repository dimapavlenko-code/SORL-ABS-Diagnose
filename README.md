# 🔧 SORL ABS Diagnostic Tool

Диагностический инструмент для блоков АБС **SORL А65R32.3538020**, устанавливаемых на коммерческие автомобили **ГАЗель NEXT** (в том числе школьные автобусы).

> **Статус проекта:** ✅ Работает. Протестировано на реальном школьном автобусе.

---

## 📖 О проекте

Этот инструмент — результат исследования проприетарного диагностического протокола блока АБС SORL, который **намеренно лишён** стандартных диагностических возможностей.

### 🏴 Манифест

Я считаю неправильным, что производитель блока АБС **скрывает от пользователей базовую информацию** о диагностике системы, которую они вынуждены эксплуатировать.

Это делается исключительно из соображений наживы — чтобы продавать за огромные деньги специализированные сканеры и диагностическое ПО. Причём речь идёт не о какой-то невероятно сложной системе — мы столкнулись с **преднамеренным усложнением жизни** рядовых пользователей, просто чтобы заставить их платить.

Реальность такова:
- Сервисов с нужным оборудованием в доступности **просто нет**
- Ехать за несколько сот километров на автобусе с горящей лампой АБС **запрещено законом** — автобус заберут на штрафстоянку, оштрафуют водителя и школу
- Это происходит **ежедневно и повсеместно** — в масштабах всей страны и мира

Всё это — потому что кому-то постоянно мало денег.

Я очень надеюсь, что наше исследование поможет всем хорошим людям, которые хотят ездить на исправном транспорте и готовы приложить собственные руки и голову. Тем более, что с помощью ИИ (в данном случае — **Qwen**) рутинные переборы проходят намного быстрее и больше похожи на игру, чем на тяжкий труд.

## 🔧 Требования

### Железо

Подходит **любой K-line → UART адаптер**. Проверены:

- ✅ Самодельный адаптер на **FT232**
- ✅ Кабель **VAG KKL** (409.1)
- ✅ Любой другой USB-K-Line адаптер на базе FT232 / CP2102 / CH340

### Подключение к автомобилю

| Параметр | Значение |
|----------|----------|
| Диагностический разъём | Стандартный **OBD-II** (16 pin) |
| Пин K-Line | **7** |
| Пин GND | **4** или **5** |
| Питание | От автомобиля (не требуется для диагностики) |

### Архитектура шины

На K-Line ГАЗели NEXT доступны **всего 2 блока**:
- ЭБУ двигателя (Jammed / Микас и т.п.)
- **ЭБУ АБС SORL** (адрес `0x28`)

> ⚠️ **Важно:** ЭБУ АБС работает **полностью автономно** и не связано с ЭБУ двигателя. Ошибки АБС не влияют на работу двигателя и наоборот.

### Диагностические интерфейсы блока SORL

| Интерфейс | Реализован? |
|-----------|-------------|
| **K-Line** | ✅ Да (единственный интерфейс) |
| CAN-шина | ❌ Нет |
| Blink-диагностика | ❌ Нет (проверены все известные методы активации) |
| UDS через K-Line | ⚠️ Частично (см. ниже) |

## 🔬 Технические детали протокола SORL


### Физический уровень

| Параметр | Значение |
|----------|----------|
| Протокол | K-Line (ISO 9141) |
| Скорость обмена | **10400 бод** |
| Параметры UART | 8N1 (8 бит, без чётности, 1 стоп-бит) |
| Адрес блока АБС | **`0x28`** |
| Адрес тестера | `0xF1` |

### 🔑 Нестандартная инициализация (САМОЕ ВАЖНОЕ!)

Блок SORL **не отвечает** на стандартную K-Line инициализацию (5 baud, `0x33 0xF9`). Вместо этого используется фирменная последовательность:

```
1. Переключить скорость UART в 200 бод
2. Отправить один байт: 0xF0
3. Подождать 500 мс
4. Переключить скорость обратно в 10400 бод
5. Отправить команду StartDiagnosticSession
```

### Формат кадра KWP2000 (с заголовком)

```
[Заголовок] [Адрес получателя] [Адрес отправителя] [Сервис] [Данные...] [Checksum]
```

**Checksum** — сумма всех байт кадра (кроме checksum) **mod 256**.

> ⚠️ В отличие от стандарта ISO 14230 (где checksum = XOR всех байт), SORL использует **простую сумму**. Это критически важно!

### Типы заголовков

| Заголовок | Описание |
|-----------|----------|
| `0x81` | Короткий кадр, сервис без данных |
| `0x82` | Короткий кадр, сервис + 1 байт данных |
| `0x83` | Короткий кадр, сервис + 2 байта данных |
| `0x84` | Короткий кадр, сервис + 3 байта данных |

## ✅ Успешные команды (подробно)

### 1. Инициализация сессии

**Отправка:**
```
200 бод → байт 0xF0 → пауза 500 мс → 10400 бод
```

**Затем команда StartDiagnosticSession:**
```
TX: 81 28 F1 81 1B
RX: 83 F1 28 C1 EA 8F D6
```

Расшифровка ответа:
- `83` — заголовок (3 байта данных)
- `F1` — адрес отправителя (блок)
- `28` — адрес получателя (тестер)
- `C1` — положительный ответ на сервис `0x81` (StartCommunication)
- `EA` — Key Byte
- `8F D6` — дополнительная информация

### 2. Открытие расширенного диагностического сеанса

```
TX: 82 28 F1 10 81 2C
RX: 82 F1 28 50 81 6C
```

- `10` — сервис StartDiagnosticSession
- `81` — расширенный режим
- `50` — положительный ответ

### 3. Поддержание сессии (Tester Present)

Необходимо отправлять каждые 1-2 секунды, иначе блок закроет сеанс:

```
TX: 81 28 F1 3E D8
RX: 81 F1 28 7E 18
```

- `3E` — сервис TesterPresent
- `7E` — положительный ответ

### 4. Чтение Local ID (идентификация блока)

Сервис `0x1A` (ReadLocalIdentifier). Примеры:

**Артикул блока (LID 0x91):**
```
TX: 82 28 F1 1A 91 46
RX: 90 F1 28 5A 91 41 36 35 52 33 32 2E 33 35 33 38 30 32 30 8A
```
ASCII: `A65R32.3538020`

**Версия ПО (LID 0x92):**
```
TX: 82 28 F1 1A 92 47
RX: 8D F1 28 5A 92 56 31 2E 30 31 31 20 4E 45 58 54 38
```
ASCII: `V1.011 NEXT`

**Тип блока (LID 0x97):**
```
TX: 82 28 F1 1A 97 4C
RX: 96 F1 28 5A 97 68 79 64 72 61 75 6C 69 63 20 41 42 53 20 38 30 28 33 38 29 9F
```
ASCII: `hydraulic ABS 80(38)`

**Счётчик ошибок (LID 0x9A):**
```
TX: 82 28 F1 1A 9A 4F
RX: 84 F1 28 5A 9A 00 02 93
```
Значение: `00 02` = 2 ошибки

### 5. Чтение дампа памяти (энергонезависимая память ошибок)

Сервис `0x21`, параметр `0x01`. Возвращает 32 байта:

```
TX: 82 28 F1 21 01 BD
RX: A2 F1 28 61 01 FF FF 00 FF FF 99 FF 00 00 00 00 FF FF FF F5 F3 C4 00 00 00 00 00 00 00 00 00 00 00 00 FF FF FF 57
```

**Структура дампа:**
- Байт `[02]` — режим работы
- Байт `[05]` — индекс/указатель ошибки (меняется при отключении датчиков)
- Байты `[0E-10]` — возможный код ошибки

> 💡 Дамп **энергонезависимый** — сохраняется при отключении питания. Меняется при отключении/подключении датчиков.

### 6. 🎯 СТИРАНИЕ ОШИБОК (главное открытие!)

Сервис `0x31` (RoutineControl) с RoutineIdentifier `0x02 0x01` — **Erase Mirror Memory DTC** по стандарту ISO 14229 (UDS).

> ⚠️ **Требует предварительного открытия расширенного сеанса!**

```
TX: 83 28 F1 31 02 01 55
RX: 83 F1 28 71 02 01 C6
```

- `31` — сервис RoutineControl
- `02 01` — RoutineIdentifier (eraseMirrorMemoryDTC)
- `71` — положительный ответ

**При успешном выполнении:**
- 🔊 Блок включает гидронасос на ~3 секунды (слышен характерный гул)
- 🔧 Проводит самодиагностику клапанов
- 💾 Очищает энергонезависимую память ошибок

### 7. Известные Local ID

| LID | Описание | Пример данных |
|-----|----------|---------------|
| `0x91` | Артикул блока | `A65R32.3538020` |
| `0x92` | Версия ПО / модель авто | `V1.011 NEXT` |
| `0x94` | Модель автомобиля | `NEXT` |
| `0x95` | Код спецификации | `ZVC.3.51` |
| `0x97` | Тип блока | `hydraulic ABS 80(38)` |
| `0x9A` | Счётчик ошибок | `00 02` |
| `0x9D` | Статус системы | `02` |
| `0xE2` | Дополнительный статус | `00` |
| `0xED` | Буфер данных | `00 00 00 00 00 00` |

## ❌ Неудачные попытки (кратко)

Для полноты картины — список подходов, которые **не сработали** (чтобы другие исследователи не тратили время):

### Стандартные сервисы чтения DTC
- `0x13` (ReadDTCByStatus) → `7F 13 11` (serviceNotSupported)
- `0x17` (ReadDTCByStatus) → `7F 17 31` (requestOutOfRange) со всеми масками 0x00-0xFF
- `0x18` (ReportDTCByStatusMask) → нет ответа
- `0x19` (ReadDTCInformation) → нет ответа
- `0x07` (ReadPendingDTC) → `7F 07 11`

### Сервисы очистки ошибок
- `0x04` (ClearDiagnosticInformation) → `7F 04 11`
- `0x14` (ClearDTC) → `7F 14 31` / `7F 14 72` (без расширенного сеанса)

### Другие подходы
- Сервис `0x22` (ReadDataByIdentifier) → `7F 22 11` со всеми DID
- Сервис `0x23` (ReadMemoryByAddress) → `7F 23 11`
- Сервис `0x24` (ReadScalingData) → `7F 24 11`
- Сервис `0x2A` (ReadDataByParameter) → `7F 2A 11`
- Сервис `0x3B` (WriteDataByLocalIdentifier) → только эхо, запись игнорируется
- Сервис `0x3C` (альтернативная запись) → только эхо
- Сервис `0x2E` (UDS WriteDataByIdentifier) → только эхо
- Сервис `0xB0` (EcuReset) → `7F B0 11`
- Перебор LID 0x00-0x8F → все `7F 1A 12` (subFunctionNotSupported)
- Перебор LID 0xA0-0xBF → все `7F 1A 12`
- Перебор LID 0xD0-0xFF → все `7F 1A 12`, кроме 0xE2 и 0xED
- Кастомные сервисы 0xA0-0xAF → все `7F xx 11`
- Чтение с других адресов (0x29-0x33) → только эхо
- Стандартная ISO 9141 инициализация (`0x68 0x6A 0xAF`) → не работает
- Blink-диагностика → не реализована в блоке
- Отключение питания на 30+ секунд → дамп не меняется (энергонезависимая память)

### Альтернативные команды Routine Control (не сработали)
- `0x31 02 02` (Erase Persistent DTC)
- `0x31 02 03` (Erase DTC Info)
- `0x31 FF 00` (общий сброс)
- `0x31 02 01 FF` (с option record)
